## Summary

---

## 1. Missing Critical Bugs

### GAP-01: Cart Store Race Condition (V5 CRIT-02) — NOT IN v2.0

**V5 reference:** `docs/V5_MILESTONE_MVP.md` → CRIT-02
**File:** `src/lib/stores/cart-store.ts:51-73`

`shouldDebounce()` reads/writes a shared global `recentAdditions` Map outside Zustand's atomic `set()` transaction. Two concurrent `addItem()` calls both pass the debounce check, allowing duplicate items.

**Recommendation:** Add to Phase 89 as BUG-06. Fix: move debounce tracking inside `set()` callback, or use React-level `useDebounce` on form submission.

**Severity:** CRITICAL — real users double-tap "Add to Cart" on mobile.

### GAP-02: Cutoff Off-by-Seconds (V5 CRIT-04) — POSSIBLY RESOLVED IN v1.9

**V5 reference:** `docs/V5_MILESTONE_MVP.md` → CRIT-04
**File:** `src/app/api/checkout/session/route.ts:57`

Millisecond-precision comparison allows orders at cutoff_hour:59:59 to pass server validation, but by the time the DB insert completes, the actual cutoff has passed. Fix is a 10-second safety buffer.

v1.9 shipped "full datetime cutoff logic" but the V5 audit was written after reviewing the v1.9 code — meaning this may still be present.

**Recommendation:** Verify `checkout/session/route.ts` cutoff logic. If no safety buffer exists, add to Phase 89 as BUG-07.

## 4. Phase-Level Feedback

### Phase 89: Critical Bug Fixes

**Strengths:** Correct priority. Dependencies: none (first phase). Success criteria are testable.

**Issues:**
- Missing CRIT-02 (cart race condition) — see GAP-01
- Missing CRIT-04 verification (cutoff buffer) — see GAP-02
- BUG-04 description says "Fix type assertion null crash on RPC checkout result" but V5 pinpoints line 283 of `api/checkout/session/route.ts`. Add the file reference.
- Consider adding BF-11 from V5 (refund ceiling) — currently MEDIUM severity but BUG-05 already covers it. Good.

**Recommendation:** Add BUG-06 (cart race condition). Verify CRIT-04 status.

### Phase 90: Menu & Photo Pipeline

**Strengths:** Good scope. Photo fallbacks already exist in `data/menu-photos/` (53/53 items covered), so this phase focuses on the admin upload workflow rather than content creation.

**Issues:**
- MENU-03 says "min 800x600" but existing platform photos vary in dimensions. Need a resize-on-upload strategy, not rejection.
- No mention of seeding fallback photos from `data/menu-photos/` into Supabase Storage for items where `image_url IS NULL`. Add a migration or seed task.
- MS-04 from V5 (Mont Ti vs Nan-Gyi Mont Ti reconciliation) is still unresolved and not in v2.0. Minor but worth tracking.

**Recommendation:** Add a task to seed `data/menu-photos/` fallbacks into Supabase Storage for items without admin photos. Add Mont Ti verification to MS-03 (owner verification).

### Phase 91: Checkout & Payment Hardening

**Strengths:** 1:1 match with V5 Phase 2. Server-side pricing (CHKT-01) is the highest-value change. Success criteria are concrete.

**Issues:**
- CHKT-09 (guest cart) is the riskiest item — anonymous cart → auth at payment touches auth, cart persistence, and Stripe session creation. Consider splitting into sub-tasks.
- V5 CP-10 (checkout success logging) maps to CHKT-10 but the v2.0 description is vague. Specify: log order ID, total_cents, user_id, payment_intent_id to Sentry breadcrumb + DB.

**Recommendation:** Break CHKT-09 into: (a) anonymous browsing + cart, (b) auth prompt at checkout, (c) cart transfer on sign-in. Add specifics to CHKT-10.

### Phase 92: Customer UX — Discovery & Shopping

**Strengths:** Practical scope. Depends on Phase 90 (photos) and 91 (checkout) — correct.

**Issues:**
- CUX-05 (Saturday hero banner) overlaps with v1.9's "dynamic hero CTA" and "menu delivery banner." Verify what already exists to avoid rebuilding.
- CUX-10 (offline banner) overlaps with v1.6's "animated offline banner" and v1.4's offline indicator. Likely needs polish, not full rebuild.
- V5 UX-06 (cutoff countdown in cart drawer) is not in v2.0 CUX requirements. v1.9 has "cart cutoff countdown" — verify completeness or add.

**Recommendation:** Audit existing v1.9 implementations of CUX-05 and CUX-10 before planning. May reduce scope.

### Phase 93: Customer UX — Engagement & Accessibility

**Strengths:** Accessibility items (CUX-14 to CUX-19) are well-specified. Engagement features (reorder, rating, sharing) add real value.

**Issues:**
- CUX-12 (rating prompt) needs backend: `ratings` table, API route, admin view. Not mentioned in success criteria.
- CUX-13 (order sharing) — V5 said "social proof" but v2.0 Out of Scope says "Social media integration — manual sharing sufficient." Clarify: is this a share link (URL copy) or social media integration?

**Recommendation:** Add backend requirements for CUX-12. Clarify CUX-13 scope (share URL vs social integration).

### Phase 94: Admin & Driver Enhancements

**Strengths:** Lean scope — only 4 requirements. Smart given that v1.8-v1.9 shipped most admin/driver features.

**Issues:**
- DRV-03 (photo proof) needs Supabase Storage bucket configuration for delivery photos, similar to menu photos. Note the dependency on Phase 90's photo infrastructure.
- ADMIN-01 (time-window grouping) — v1.9 shipped the ops dashboard with 5s polling and bulk status changes. This adds grouping on top. Clarify: is this a UI reorganization or new data model?

**Recommendation:** Note DRV-03's dependency on photo storage infrastructure from Phase 90.

### Phase 95: Observability, Performance & Testing

**Strengths:** Capstone phase with dry run + load test is the right call. 14 requirements cover the full stack.

**Issues:**
- OBS-04 (daily DB backup) — Supabase Pro has automated backups. If on Pro plan, this is configuration, not code. If on free tier, needs custom solution.
- TST-06 (Saturday dry run with 20 orders) needs a test data strategy: test Stripe keys, test users, test addresses. Add a pre-dry-run setup task.
- TST-07 (50 concurrent checkouts) needs a load testing tool decision: k6, Artillery, or custom script. Worth noting in plans.
- OBS-06 (bundle under 200KB) — v1.5 got LCP from 19.9s to 10.9s but current first-load JS is likely >200KB. This may require significant tree-shaking. Assess feasibility before committing.

**Recommendation:** Add dry-run setup task. Decide load testing tool. Verify bundle target is realistic.

## 5. Structural Recommendations

### Add a Pre-Launch Checklist Phase

V5 had a detailed pre-launch checklist (infrastructure, verification, operations) that isn't in v2.0. Consider adding it as Phase 95.5 or appending to Phase 95:

- [ ] Supabase production instance (separate from staging)
- [ ] Production env vars (Stripe live keys, Resend domain, Sentry DSN)
- [ ] DNS + custom domain
- [ ] Google Maps API billing with budget cap
- [ ] Upstash Redis provisioned
- [ ] SSL certificate valid
- [ ] Stripe webhook tested with real test payments
- [ ] Email delivery confirmed (all 4 templates)
- [ ] Mobile testing (iOS Safari, Android Chrome, PWA)
- [ ] Admin trained on ops dashboard
- [ ] Driver(s) completed test deliveries
- [ ] Refund process documented
- [ ] Emergency procedures documented

### Add V5 Risk Register to v2.0

V5 identified 8 risks with mitigations. These should live in v2.0 planning:

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Stripe webhook failures lose orders | Medium | Critical | Webhook retry + reconciliation |
| Menu photos not available from owner | High | Medium | Fallbacks exist in `data/menu-photos/` |
| Driver app offline during delivery | Medium | High | IndexedDB queue built; test in Phase 95 |
| Cutoff time confusion | High | Medium | Phase 92 hero banner + countdown |
| Google Maps API quota | Low | High | Budget caps + Leaflet fallback |
| Supabase free tier limits | Medium | Critical | Monitor; upgrade before launch |
| Platform price drift | High | High | Periodic manual check; automated sync deferred |
| Pricing strategy unclear | Medium | Medium | App = direct pricing; platforms add ~30% |

### Add Success Metrics

V5 defined post-launch success metrics. These should be in v2.0 acceptance criteria:

| Metric | Target |
|--------|--------|
| Orders per Saturday | 30+ |
| Checkout completion rate | >70% |
| Average order value | >$40 |
| Zero payment double-charges | 100% |
| Page load time (LCP) | <2.5s |
