**Tabs.tsx -- Replace layoutId pill with CSS-positioned indicator**

The current pattern: each tab renders a conditional `<m.div layoutId={layoutId}>` when active. The layoutId causes the pill to animate between tab positions.

New pattern: Render ONE indicator `<div>` that uses CSS `transition` and `style={{ left, width }}` to animate position.

1. Add a `tabRefs` map using `useRef<Map<string, HTMLButtonElement>>()` to track each tab button's position.
2. Add state for indicator position: `const [indicatorStyle, setIndicatorStyle] = useState<{ left: number; width: number } | null>(null);`
3. Add a `useEffect` that runs when `activeTab` changes:
   - Gets the active tab button from `tabRefs`
   - Reads its `offsetLeft` and `offsetWidth`
   - Sets `indicatorStyle` to `{ left: offsetLeft, width: offsetWidth }`
4. Add a `ResizeObserver` to recalculate on resize.
5. Replace the per-tab conditional `{isActive && (<m.div layoutId={layoutId} .../>)}` with a single indicator div rendered once outside the tab loop but inside the scroll container:
   ```tsx
   {indicatorStyle && (
     <div
       className="absolute top-1 bg-surface-primary rounded-input shadow-sm transition-all duration-200 ease-out"
       style={{
         left: indicatorStyle.left,
         width: indicatorStyle.width,
         height: 'calc(100% - 8px)',
       }}
     />
   )}
   ```
6. Remove the `layoutId` prop from the `TabsProps` interface and the component props. Keep it in the interface as `@deprecated` OR remove entirely -- it's no longer used since CSS handles the animation.

   **Decision:** Remove `layoutId` prop entirely. Any consumers passing `layoutId` will get a TypeScript error guiding them to update. Check `layoutId` usage:
   - `Tabs` consumers set `layoutId` as a prop -- after removing from interface, `pnpm typecheck` will flag them
   - Fix each consumer by removing the `layoutId` prop from their `<Tabs ... />` JSX

7. Remove `m` import from "framer-motion" IF no other `m.` usage remains. The component may still use `m.div` for something -- check. Current code only uses `m.div` for the layoutId pill. After removing it, `m` import can be dropped.

**BottomNav.tsx -- Replace layoutId indicator with CSS transition**

Current pattern: Each nav link renders `{active && <m.span layoutId="bottomNavIndicator" />}`.

New pattern: Single indicator span positioned with CSS transition.

1. Calculate active item index from `pathname` matching.
2. Render one `<span>` with `transition-all duration-200 ease-out` and `style={{ left: calculated position }}`:
   - Since nav items are evenly distributed (`justify-around` in a flex with 4 items), position = `(index * 25%) + 12.5% - 12px` (center of each quarter)
   - Width is fixed at `w-6` (24px), height `h-0.5`
   - Use `transform: translateX(...)` based on active index for smooth repositioning
3. Alternative simpler approach: keep the indicator per-item but replace `m.span` with plain `<span>` and use CSS transition on the parent. Since only one indicator exists at a time (conditional render), use an absolutely positioned span that transitions via `left` based on active index.

**Simplest approach for BottomNav:** Since items are evenly spaced:
```tsx
const activeIndex = items.findIndex(item => isActive(item.href));
```
Render one indicator at the bottom of the nav:
```tsx
<span
  className="absolute bottom-1 h-0.5 w-6 rounded-full bg-primary transition-all duration-200 ease-out"
  style={{
    left: `calc(${(activeIndex / items.length) * 100}% + ${100 / items.length / 2}% - 12px)`,
    opacity: activeIndex >= 0 ? 1 : 0,
  }}
/>
```

4. Remove `layoutId="bottomNavIndicator"` and the per-item conditional indicator.
5. Keep `m.span` for the icon scale animation (`animate: { scale: active ? 1.1 : 1 }`) -- this works with domAnimation.
6. Remove `m` import ONLY if the icon scale animation is the only remaining `m.` usage -- check. The icon uses `m.span` with `animate` -- keep `m` import.
