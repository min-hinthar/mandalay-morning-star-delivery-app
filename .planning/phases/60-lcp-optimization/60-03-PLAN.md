**Step 1: Create DomMaxProvider component**

Create `src/components/providers/DomMaxProvider.tsx`:

```typescript
"use client";

import type { ReactNode } from "react";
import { LazyMotion } from "framer-motion";

const loadDomMax = () =>
  import("framer-motion").then((mod) => mod.domMax);

interface DomMaxProviderProps {
  children: ReactNode;
}

/**
 * Async domMax feature provider for routes needing layout animations, drag, or useAnimate.
 *
 * Wraps children with LazyMotion loading domMax asynchronously.
 * Nests inside the root domAnimation provider -- inner LazyMotion overrides outer.
 *
 * Use on: customer routes (cart drag, layoutId), admin routes (layoutId),
 * driver routes (layoutId), auth routes (layoutId, useAnimate).
 */
export function DomMaxProvider({ children }: DomMaxProviderProps) {
  return (
    <LazyMotion features={loadDomMax} strict>
      {children}
    </LazyMotion>
  );
}
```

Check if `src/components/providers/` directory exists. If not, create it. Check existing files in that path to avoid conflicts.

**Step 6: Handle global Toast drag**

Toast uses `drag="x"` and is rendered globally via `ToastProvider` in root layout. It needs domMax for drag.

Options:
a) Wrap just ToastProvider with DomMaxProvider in root layout
b) Replace Toast drag with touch event handlers (removes domMax need)
c) Accept Toast without drag on homepage, drag works on routes with DomMaxProvider

**Decision (a):** Wrap ToastProvider with DomMaxProvider in root layout. This adds an async domMax load globally BUT it's async (non-blocking), so it doesn't affect LCP. The root layout already has `<Providers>` with async domAnimation. Adding a nested DomMaxProvider around ToastProvider means:
- Root: domAnimation (async) -- for all pages
- ToastProvider: domMax (async) -- just for toast drag
- Route layouts: domMax (async) -- for route-specific features

The ToastProvider nesting means toast gets domMax while the rest of the page gets domAnimation from root. This works because LazyMotion nesting is supported.

In `src/app/layout.tsx`, wrap the ToastProvider:

**Wait -- this would make DomMaxProvider wrap ALL children including the page content, defeating the purpose.**

**Revised approach:** Do NOT wrap children with DomMaxProvider globally. Instead:
- Option (c): Toast drag only works on routes with DomMaxProvider (customer, admin, driver, auth). On public routes, toast renders and dismisses via the X button but drag doesn't animate. This is acceptable -- toast content and dismiss button always work; drag is a nice-to-have.
- OR: Wrap ONLY the ToastProvider element, not children. But ToastProvider wraps children in its implementation.

Read `ToastProvider` to understand its structure. If ToastProvider renders a portal (toast outside React tree), then drag may work if the portal is inside a DomMaxProvider context.

Actually, `Toast.tsx` uses `<Portal>` component. The Portal renders outside the main React tree. So even if we wrap something with DomMaxProvider, the Portal content won't be inside that context.

**Final decision:** Accept that Toast drag won't animate on public routes (it will still be dismissible via X button). On customer/admin/driver/auth routes, Toast portal will pick up the closest LazyMotion context from the root providers.tsx (domAnimation). Drag needs domMax which isn't available globally.

**Better alternative:** Refactor Toast dismiss to use a simple click handler and remove `drag="x"` entirely. Replace with CSS swipe or just keep the X button as primary dismiss. This is the cleanest solution -- Toast drag is a minor polish, not a signature animation.

**Decision: Remove drag from Toast.** Per user context: "signature animations to preserve: cart bounce/spring, page transitions, all playful animation identity." Toast swipe-to-dismiss is not listed as signature. Replace `drag="x"` with a simple click/tap dismiss. Keep the enter/exit AnimatePresence animations (work with domAnimation).
