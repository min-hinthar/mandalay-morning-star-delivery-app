## 3. Database Schema Analysis

### Drivers Table

```sql
drivers (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES profiles,
  vehicle_type vehicle_type,  -- 'car', 'motorcycle', 'bicycle', 'van', 'truck'
  license_plate TEXT,
  phone TEXT,
  profile_image_url TEXT,
  is_active BOOLEAN DEFAULT true,
  onboarding_completed_at TIMESTAMPTZ,
  rating_avg NUMERIC(3,2),
  deliveries_count INTEGER DEFAULT 0
)
```

### Routes Table

```sql
routes (
  id UUID PRIMARY KEY,
  delivery_date DATE,
  driver_id UUID REFERENCES drivers,
  status route_status,  -- 'planned', 'in_progress', 'completed'
  optimized_polyline TEXT,
  stats_json JSONB,
  started_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ
)
```

### Route Stops Table

```sql
route_stops (
  id UUID PRIMARY KEY,
  route_id UUID REFERENCES routes,
  order_id UUID REFERENCES orders,
  stop_index INTEGER,
  eta TIMESTAMPTZ,
  status route_stop_status,  -- 'pending', 'enroute', 'arrived', 'delivered', 'skipped'
  arrived_at TIMESTAMPTZ,
  delivered_at TIMESTAMPTZ,
  delivery_photo_url TEXT,
  delivery_notes TEXT
)
```

### Delivery Exceptions Table

```sql
delivery_exceptions (
  id UUID PRIMARY KEY,
  route_stop_id UUID REFERENCES route_stops,
  exception_type delivery_exception_type,  -- 'customer_not_home', 'wrong_address', etc.
  description TEXT,
  photo_url TEXT,
  resolved_at TIMESTAMPTZ,
  resolved_by UUID REFERENCES profiles,
  resolution_notes TEXT
)
```

### Driver Ratings Table

```sql
driver_ratings (
  id UUID PRIMARY KEY,
  driver_id UUID REFERENCES drivers,
  order_id UUID REFERENCES orders,
  route_stop_id UUID REFERENCES route_stops,
  rating INTEGER CHECK (1-5),
  feedback_text TEXT,
  submitted_at TIMESTAMPTZ
)
```

Need to create `app_settings` table for storing configurable settings.

## 4. Google Maps Integration

### Current Implementation

**Package:** `@react-google-maps/api` v2.20.8 (already installed)

### Existing Map Components

| Component          | Location                                            | Features                         |
| ------------------ | --------------------------------------------------- | -------------------------------- |
| `DeliveryMap`      | `src/components/ui/orders/tracking/DeliveryMap.tsx` | Live tracking, polyline, markers |
| `CoverageRouteMap` | `src/components/ui/coverage/CoverageRouteMap.tsx`   | Coverage circles, route preview  |

### Map Patterns Established

- `useJsApiLoader` for loading
- Libraries: `['places', 'geometry', 'marker']`
- Custom warm map styles matching brand
- `AdvancedMarkerElement` with Map ID fallback
- Polyline decoding via `google.maps.geometry.encoding.decodePath`
- Fullscreen expansion with body scroll lock
- Mobile crash prevention: no backdrop-blur on mobile Safari

### Environment Variables

```
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY  # Client-side
GOOGLE_MAPS_API_KEY              # Server-side (Routes API)
NEXT_PUBLIC_GOOGLE_MAPS_MAP_ID   # For AdvancedMarkerElement
```

### Route Detail Map Requirements

- Large map showing route polyline
- Numbered stop markers (1, 2, 3...)
- Different marker colors for status (pending=blue, delivered=green, exception=red)
- Click marker to scroll to stop card
- Fit bounds to show all stops
- Optional live driver position overlay when in_progress

## 10. Technical Risks & Mitigations

### Risk: Google Maps Performance

- **Issue:** Multiple markers + polyline + live updates can lag
- **Mitigation:** Use `AdvancedMarkerElement`, debounce live updates, virtualize stop list

### Risk: Settings Cache Invalidation

- **Issue:** Settings changed but not reflected
- **Mitigation:** TanStack Query with refetchOnWindowFocus, or Supabase realtime

### Risk: Order Edit Conflicts

- **Issue:** Admin edits while customer views
- **Mitigation:** Optimistic locking (check updated_at before save)

### Risk: Large Route Maps

- **Issue:** Routes with 15+ stops overwhelming the map
- **Mitigation:** Fit bounds with padding, cluster markers if needed
