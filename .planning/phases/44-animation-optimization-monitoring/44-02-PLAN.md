  <name>Task 1: Add LazyMotion provider to application root</name>
  <files>src/app/providers.tsx</files>
  <action>
    1. Edit `src/app/providers.tsx` to wrap with LazyMotion:

       ```typescript
       "use client";

       import type { ReactNode } from "react";
       import { LazyMotion, domMax } from "framer-motion";
       import { QueryProvider } from "@/lib/providers/query-provider";
       import { AnimationProvider } from "@/lib/providers/animation-provider";
       import { ThemeProvider, DynamicThemeProvider } from "@/components/ui/theme";

       interface ProvidersProps {
         children: ReactNode;
       }

       export function Providers({ children }: ProvidersProps) {
         return (
           <ThemeProvider>
             <DynamicThemeProvider>
               <QueryProvider>
                 <LazyMotion features={domMax} strict>
                   <AnimationProvider>
                     {children}
                   </AnimationProvider>
                 </LazyMotion>
               </QueryProvider>
             </DynamicThemeProvider>
           </ThemeProvider>
         );
       }
       ```

    Key decisions:
    - `domMax` (not `domAnimation`) because codebase uses `drag` (CartItem.tsx) and `layoutId` (~10 components)
    - Synchronous loading (not async) because animations appear on every page above-the-fold
    - Root level placement (in providers.tsx) so ALL routes get animation features
    - `strict` mode to enforce m.* usage and catch any remaining motion.* at runtime
    - Placed ABOVE AnimationProvider so animation context has access to features

    NOTE: Adding `strict` before completing the migration in Task 2 will cause runtime errors for any motion.* components. This is intentional -- it makes the build/test step in Task 3 catch any missed conversions. If you prefer, add strict AFTER Task 2's migration is complete.

    Do NOT:
    - Use `domAnimation` (missing drag + layout features the app needs)
    - Use async loading pattern (would flash unanimated content)
    - Place LazyMotion below AnimatePresence usage points
    - Import from `"motion/react-m"` (known bug #3091)
  </action>
  <verify>
    File contains `LazyMotion features={domMax} strict`.
    `pnpm typecheck` passes (imports resolve correctly).
  </verify>
  <done>LazyMotion provider wraps the application with domMax features and strict mode at the root level in providers.tsx.</done>

  <action>
    This task uses automated shell commands (sed/perl) to perform the entire migration in bulk.
    Do NOT open and edit files individually. Run these commands sequentially.

    **Step 1: Snapshot the baseline count for before/after comparison**
    ```bash
    echo "=== BEFORE MIGRATION ==="
    echo "Files with motion. JSX:" $(grep -rl "<motion\." src/ --include="*.tsx" | wc -l)
    echo "Total motion. JSX occurrences:" $(grep -r "<motion\.\|</motion\." src/ --include="*.tsx" | wc -l)
    echo "Files with motion import:" $(grep -rl "import.*{.*motion.*}.*from.*framer-motion" src/ --include="*.tsx" --include="*.ts" | wc -l)
    ```

    **Step 2: Replace all motion.* JSX elements with m.* (opening and self-closing tags)**
    ```bash
    # Replace <motion. with <m. in all tsx files
    find src/ -name "*.tsx" -exec sed -i 's/<motion\./<m./g' {} +

    # Replace </motion. with </m. in all tsx files (closing tags)
    find src/ -name "*.tsx" -exec sed -i 's/<\/motion\./<\/m./g' {} +
    ```

    **Step 3: Replace motion import with m in import statements**
    This requires careful regex to only replace `motion` when it's a named import from framer-motion, not other uses.
    ```bash
    # Handle: import { motion } from "framer-motion"
    # Handle: import { motion, ... } from "framer-motion"
    # Handle: import { ..., motion } from "framer-motion"
    # Handle: import { ..., motion, ... } from "framer-motion"
    # Uses perl for non-greedy matching and lookahead
    find src/ \( -name "*.tsx" -o -name "*.ts" \) -exec perl -pi -e '
      # Only operate on lines that import from framer-motion
      if (/from\s+["\x27]framer-motion["\x27]/) {
        # Replace "motion" as a standalone import name (word boundary)
        # Avoid replacing "useMotionValue", "useMotionValueEvent", etc.
        s/\bmotion\b(?![-_A-Za-z])/m/g;
      }
    ' {} +
    ```

    **Step 4: Replace type references (typeof motion.* -> typeof m.*)**
    ```bash
    find src/ \( -name "*.tsx" -o -name "*.ts" \) -exec sed -i 's/typeof motion\./typeof m./g' {} +
    ```

    **Step 5: Fix any over-replacements**
    The perl regex in Step 3 should be precise, but verify and fix these known safe patterns:
    ```bash
    # Check that "framer-motion" package name was NOT corrupted
    # (it shouldn't be since we match word boundary \b, but verify)
    grep -rn '"framer-m"' src/ --include="*.tsx" --include="*.ts"
    # If any results: sed -i 's/"framer-m"/"framer-motion"/g' on those files

    # Check that motion-tokens references were NOT corrupted
    grep -rn 'm-tokens' src/ --include="*.tsx" --include="*.ts"
    # If any results: sed -i 's/m-tokens/motion-tokens/g' on those files

    # Check that @/lib/motion paths were NOT corrupted
    grep -rn '@/lib/m"' src/ --include="*.tsx" --include="*.ts"
    # If any results: sed -i 's/@\/lib\/m"/@\/lib\/motion"/g' on those files

    # Check useMotionValue, useMotionValueEvent were NOT corrupted
    grep -rn 'useMValue\|useMValueEvent\|useMTemplate' src/ --include="*.tsx" --include="*.ts"
    # These should return zero — the \b boundary and negative lookahead prevent this
    ```

    **Step 6: Snapshot the after count**
