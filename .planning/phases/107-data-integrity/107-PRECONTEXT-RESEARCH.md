## 7. Architectural Decisions

### Decision 1: FOR UPDATE SKIP LOCKED vs simple WHERE guard

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **FOR UPDATE SKIP LOCKED** (RPC) | Atomic; prevents double-selection; standard PostgreSQL pattern | Requires new migration; slightly more complex SQL | **CHOSEN** |
| WHERE clause guard (`UPDATE ... WHERE status='pending'`) | Simpler; no locking overhead | Two requests still SELECT same stop; one UPDATE succeeds, other is no-op but client gets stale data | Rejected |
| Optimistic locking (version column) | Application-level; no SQL changes | Adds schema complexity; overkill for this use case | Rejected |

**Rationale:** SKIP LOCKED prevents even the selection of a locked row, ensuring two concurrent requests always get different stops. WHERE guard alone still has the TOCTOU window between SELECT and UPDATE.

### Decision 2: RPC return type

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **RETURNS jsonb** | Flexible; can include stop_id + stop_index | Requires JSON parsing in TypeScript | **CHOSEN** |
| RETURNS uuid | Simple; just the promoted stop ID | Loses stop_index info needed by client | Rejected |
| RETURNS void | Simplest | Client can't know which stop was promoted | Rejected |

### Decision 3: Where to call updateRouteStats

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **Inside the RPC** (PERFORM update_route_stats) | Atomic with promotion | Tighter coupling | **CHOSEN** |
| Keep in TypeScript after RPC call | Looser coupling | Non-atomic; stats may not reflect promotion | Rejected |

## 12. Proposed RPC Implementation

```sql
CREATE OR REPLACE FUNCTION promote_next_stop(
  p_route_id uuid,
  p_completed_stop_id uuid
)
RETURNS jsonb AS $$
DECLARE
  v_next_stop_id uuid;
  v_next_stop_index int;
BEGIN
  -- Validate completed stop belongs to this route and is terminal
  IF NOT EXISTS (
    SELECT 1 FROM route_stops
    WHERE id = p_completed_stop_id
      AND route_id = p_route_id
      AND status IN ('delivered', 'skipped')
  ) THEN
    RAISE EXCEPTION 'Stop % is not a completed stop in route %',
      p_completed_stop_id, p_route_id
      USING ERRCODE = 'invalid_parameter_value';
  END IF;

  -- Atomically select and lock the next pending stop
  -- SKIP LOCKED: if another transaction has this row locked, skip it
  -- and find the one after (prevents double-promotion)
  SELECT id, stop_index
  INTO v_next_stop_id, v_next_stop_index
  FROM route_stops
  WHERE route_id = p_route_id
    AND status = 'pending'
  ORDER BY stop_index ASC
  LIMIT 1
  FOR UPDATE SKIP LOCKED;

  -- If no pending stop found, return null (all stops complete)
  IF v_next_stop_id IS NULL THEN
    RETURN jsonb_build_object(
      'promoted_stop_id', NULL,
      'stop_index', NULL
    );
  END IF;

  -- Promote the locked stop to enroute
  UPDATE route_stops
  SET status = 'enroute'
  WHERE id = v_next_stop_id;

  -- Update route stats atomically
  PERFORM update_route_stats(p_route_id);

  RETURN jsonb_build_object(
    'promoted_stop_id', v_next_stop_id,
    'stop_index', v_next_stop_index
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER SET search_path = public;
```

## 14. Stop Completion Handler Changes

### Current (lines 165-187)
```typescript
// SELECT + UPDATE with race window
let nextStop = null;
if (newStatus === "delivered" || newStatus === "skipped") {
  const { data: nextStopData } = await supabase
    .from("route_stops")
    .select("id, stop_index")
    .eq("route_id", routeId)
    .eq("status", "pending")
    .order("stop_index", { ascending: true })
    .limit(1)
    .single();

  if (nextStopData) {
    await supabase.from("route_stops").update({ status: "enroute" }).eq("id", nextStopData.id);
    nextStop = { id: nextStopData.id, stopIndex: nextStopData.stop_index };
  }
}
```

### Proposed (atomic RPC call)
```typescript
let nextStop = null;
if (newStatus === "delivered" || newStatus === "skipped") {
  const { data: promotionResult, error: promotionError } = await supabase.rpc(
    "promote_next_stop",
    { p_route_id: routeId, p_completed_stop_id: stopId }
  );

  if (promotionError) {
    logger.warn("Stop promotion failed", {
      api: "driver/routes/[routeId]/stops/[stopId]",
      routeId, stopId, error: promotionError.message,
    });
  } else if (promotionResult?.promoted_stop_id) {
    nextStop = {
      id: promotionResult.promoted_stop_id,
      stopIndex: promotionResult.stop_index,
    };
  }
}
```

### Key Changes
1. Single atomic RPC call replaces SELECT+UPDATE
2. `updateRouteStats` moved inside RPC (removed from TypeScript)
3. Promotion failure is non-blocking (matches existing pattern)
4. Response shape unchanged ŌĆö client compatibility preserved
