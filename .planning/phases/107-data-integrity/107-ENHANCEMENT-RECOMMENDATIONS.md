## Detailed Recommendations

### 1. Atomic Stop Promotion RPC (MUST-HAVE)

**What:** Create `promote_next_stop(p_route_id, p_completed_stop_id)` PostgreSQL RPC using `FOR UPDATE SKIP LOCKED` to prevent double-promotion race condition.

**Why:** Current stop completion handler (lines 165-187 of `stops/[stopId]/route.ts`) uses a 2-step SELECT+UPDATE pattern with a race window. Two concurrent stop completions can both select the same next pending stop and both promote it to `enroute`. While rare at 20-50 orders, this is a data integrity bug that causes incorrect route state.

**Design compliance:** Follows existing RPC patterns: `SECURITY DEFINER SET search_path = public`, `p_` parameter naming, `v_` variable naming, `CREATE OR REPLACE FUNCTION` for idempotency. Matches `split_route`/`merge_routes` atomicity model.

**Implementation hint:**
- Migration: `supabase/migrations/20260321_atomic_stop_promotion.sql`
- SQL: `SELECT ... FOR UPDATE SKIP LOCKED` → `UPDATE ... SET status = 'enroute'` → `PERFORM update_route_stats(p_route_id)` — all in one function
- TypeScript: Replace lines 165-187 with single `supabase.rpc("promote_next_stop", {...})` call
- Return: `jsonb_build_object('promoted_stop_id', id, 'stop_index', idx)` for client compat

### 2. Dead Code Removal: increment_driver_deliveries (MUST-HAVE)

**What:** Remove the `supabase.rpc("increment_driver_deliveries", ...)` call and its try/catch wrapper from `complete/route.ts` lines 108-118.

**Why:** This RPC was never defined as a SQL function. The try/catch silently swallows the error with a log message "RPC not available, skipping." The actual delivery counting is handled by the `update_driver_deliveries_count` trigger which fires per-stop on `status='delivered'`. If this RPC were ever created, it would double-count deliveries.

**Design compliance:** Eliminates dead code and prevents future double-counting risk. Aligns with project's "no silent failures" principle (CLAUDE.md gotcha: "Webhook handlers: return 500 on DB errors for retry; never swallow into 200").

**Implementation hint:**
- Delete lines 108-118 from `src/app/api/driver/routes/[routeId]/complete/route.ts`
- No migration needed (RPC was never created as SQL)
- Remove `increment_driver_deliveries` type from `database.ts` Functions section (recommendation #5)

### 3. updateRouteStats Atomicity (SHOULD-HAVE)

**What:** Move `updateRouteStats` call from TypeScript (line 163 of stop handler) into the `promote_next_stop` RPC via `PERFORM update_route_stats(p_route_id)`.

**Why:** Currently, route stats are updated as a separate TypeScript call BEFORE stop promotion. If promotion fails, stats are already updated but don't reflect the promotion. Moving it inside the RPC ensures stats are always consistent with the actual stop states — both the delivered/skipped stop AND the newly-promoted enroute stop are reflected atomically.

**Design compliance:** Matches `split_route` and `merge_routes` patterns which call `PERFORM update_route_stats()` inside the RPC transaction.

**Implementation hint:**
- Add `PERFORM update_route_stats(p_route_id);` after the `UPDATE route_stops SET status = 'enroute'` in the RPC
- Remove the TypeScript `await updateRouteStats(supabase, routeId)` call from the stop handler (or keep it only for non-delivery status changes like `arrived`)
- Note: The TypeScript `updateRouteStats` function can remain for the `arrived` status path where no promotion occurs

### 4. Idempotent Promotion Guard (SHOULD-HAVE)

**What:** Add a guard in the RPC to gracefully handle cases where the completed stop is not actually in a terminal state (e.g., network retry of an already-processed request).

**Why:** If a client retries a stop completion due to network timeout, the stop may already be `delivered`. The RPC should check this and return null gracefully rather than raising an exception. This prevents 500 errors on legitimate retries.

**Design compliance:** Follows the existing idempotency pattern in the stop exception handler (commit 8ca52925) which returns 200 on duplicate exception submissions.

**Implementation hint:**
```sql
-- At start of RPC, check if stop is actually terminal
IF NOT EXISTS (
  SELECT 1 FROM route_stops
  WHERE id = p_completed_stop_id AND route_id = p_route_id
    AND status IN ('delivered', 'skipped')
) THEN
  -- Graceful return instead of RAISE EXCEPTION
  RETURN jsonb_build_object('promoted_stop_id', NULL, 'stop_index', NULL);
END IF;
```

### 6. Promotion Failure Logging (SHOULD-HAVE)

**What:** Add structured logging when stop promotion RPC returns an error (non-blocking), including route_id, stop_id, driver_id, and error details.

**Why:** Stop promotion failures are currently invisible if the RPC errors. The existing inline code had no error handling on the UPDATE. Adding structured logging enables debugging without blocking the API response.

**Design compliance:** Follows the Sentry breadcrumb pattern from Phase 105 (admin overrides logged via `captureMessage`).

**Implementation hint:**
```typescript
if (promotionError) {
  logger.warn("Stop promotion failed", {
    api: "driver/routes/[routeId]/stops/[stopId]",
    routeId, stopId, driverId,
    error: promotionError.message,
  });
}
```

### 7. Completion Rate Formula Consistency (NICE-TO-HAVE)

**What:** Align the completion rate formula between the stop handler and route completion handler.

**Why:** Stop handler `updateRouteStats` calculates `completion_rate = (delivered + skipped) / total * 100`. Route completion handler calculates `completion_rate = delivered / total * 100` (excludes skipped). This inconsistency means the rate jumps when a route is completed.

**Design compliance:** Data consistency principle — same metric should have the same formula everywhere.

**Implementation hint:**
- In `complete/route.ts` lines ~88-89, change to include skipped:
  `Math.round(((stats.delivered_stops + stats.skipped_stops) / stats.total_stops) * 100)`
- Or document the intentional difference (completion = all terminal vs delivery = only delivered)

### 8. Remove Dead calculate_route_stats Function (NICE-TO-HAVE)

**What:** Drop `calculate_route_stats()` from `002_functions.sql` — it's superseded by `update_route_stats()` in the pipeline hardening migration.

**Why:** Two functions computing the same stats creates confusion. `calculate_route_stats` (002_functions.sql:145-164) returns stats but doesn't update the route. `update_route_stats` (20260312:92-118) does both — and is the one actually called everywhere.

**Design compliance:** Dead code elimination principle from CLAUDE.md.

**Implementation hint:**
- Create migration: `DROP FUNCTION IF EXISTS calculate_route_stats(uuid);`
- Verify no call sites exist (grep confirms none)
- Risk: If any rollback chain depends on `002_functions.sql` being re-applicable, this could break. Low risk since `CREATE OR REPLACE` is idempotent.

### 9. Stop Handler updateRouteStats Dedup (NICE-TO-HAVE)

**What:** Remove the TypeScript `updateRouteStats()` call for `delivered`/`skipped` status paths since the RPC handles it atomically. Keep it only for `arrived`/`enroute` paths.

**Why:** After moving stats update into the RPC (recommendation #3), calling it again in TypeScript is redundant for delivery/skip paths. Removing the duplicate call saves one database round-trip per stop completion.

**Design compliance:** DRY principle; single source of truth for stats updates.

**Implementation hint:**
```typescript
// Only call TypeScript updateRouteStats for non-terminal status changes
if (newStatus !== "delivered" && newStatus !== "skipped") {
  await updateRouteStats(supabase, routeId);
}
```

### 10. Badge deliveries_count Freshness Check (NICE-TO-HAVE)

**What:** In route completion handler, re-fetch `drivers.deliveries_count` AFTER route completion instead of before, to ensure badge check uses the trigger-updated count.

**Why:** Current code (lines 122-132) fetches `deliveries_count` before any stop triggers may have fully propagated. In practice this is fine because triggers fire during individual stop updates (before route completion), but re-fetching after completion would be more defensive.

**Design compliance:** Phase 105 pattern: "verify affected rows with `.select()` chain."

**Implementation hint:**
- Move the `drivers` SELECT query (line 124) to after the route status UPDATE
- Or verify via testing that trigger count is already correct at this point

### 11. Route Completion Pending Stop Guard (NICE-TO-HAVE)

**What:** Add a check in the route completion TypeScript handler to verify no stops remain in `pending` or `enroute` status before marking the route complete.

**Why:** The DB trigger `check_route_completion` already enforces this, but a TypeScript guard would provide a better error message (400 "Cannot complete route: 3 stops still pending") vs the raw DB trigger error.

**Design compliance:** User-facing error quality; defense-in-depth.

**Implementation hint:**
```typescript
const pendingStops = stopCounts?.filter(s => s.status === "pending" || s.status === "enroute");
if (pendingStops && pendingStops.length > 0) {
  return NextResponse.json(
    { error: `Cannot complete route: ${pendingStops.length} stops still pending` },
    { status: 400 }
  );
}
```

### 12. Stop Promotion Sentry Breadcrumb (NICE-TO-HAVE)

**What:** Add a Sentry breadcrumb when a stop is promoted, recording the route_id, promoted stop_id, and driver_id.

**Why:** Creates an audit trail for stop progression. If a driver reports incorrect stop ordering, the breadcrumb trail shows exactly which stops were promoted and when. Follows the Phase 105 Sentry audit pattern for admin overrides.

**Design compliance:** Observability principle from Phase 105 (Sentry audit on state changes).

**Implementation hint:**
```typescript
import { addBreadcrumb } from "@sentry/nextjs";

if (nextStop) {
  addBreadcrumb({
    category: "route.stop_promotion",
    message: `Promoted stop ${nextStop.id} to enroute`,
    level: "info",
    data: { routeId, completedStopId: stopId, promotedStopId: nextStop.id },
  });
}
```
