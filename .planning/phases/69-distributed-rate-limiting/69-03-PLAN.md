    **A. Create `src/lib/hooks/useRateLimitToast.ts`:**
    - This provides a utility function for handling 429 responses in client-side fetch calls
    - Export function `handleRateLimitResponse(response: Response, context?: { isOrderPlacement?: boolean }): boolean`
      - If `response.status !== 429`, return `false` (not rate limited)
      - If rate limited:
        - If `context?.isOrderPlacement` is true, show toast: `{ variant: "warning", title: "Please wait", description: "Your order is being processed. Please don't submit again." }`
        - Otherwise, show toast: `{ variant: "destructive", title: "Too many requests", description: "Please wait a moment and try again." }`
        - Return `true` (was rate limited, caller should stop processing)
    - This is NOT a React hook (no useState/useEffect) -- it's a utility function that can be called from event handlers. Name the file `useRateLimitToast.ts` to follow the project convention of prefixing with `use` for toast-related utilities, or alternatively name it `rate-limit-toast.ts`. Check how existing toast utilities are named and follow that pattern.
    - Import `toast` from the existing toast system (`@/lib/hooks/useToast` or wherever `toast()` is exported from)

    **B. Update or create `src/lib/utils/api-client.ts` (if it exists):**
    - Check if the project has a centralized fetch wrapper. If yes, add 429 handling there.
    - If no centralized wrapper exists, create a lightweight one:
      ```typescript
      import { handleRateLimitResponse } from "@/lib/hooks/useRateLimitToast";

      export async function apiFetch(url: string, options?: RequestInit & { isOrderPlacement?: boolean }): Promise<Response> {
        const response = await fetch(url, options);
        const { isOrderPlacement, ...fetchOptions } = options ?? {};
        if (handleRateLimitResponse(response, { isOrderPlacement })) {
          throw new Error("Rate limited");
        }
        return response;
      }
      ```
    - IMPORTANT: Do NOT refactor all existing fetch calls to use this wrapper. That is out of scope. Just create the utility so it's available. The existing toast-based error handling in components will naturally show errors for most cases already.

      - After the fetch response, check if `response.status === 429`
      - If so, show the reassuring toast and prevent further processing
      - This is a targeted change, not a global refactor
