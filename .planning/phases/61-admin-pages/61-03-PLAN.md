**1. StatusChangeDialog** (`StatusChangeDialog.tsx`):

Full confirmation dialog for status changes per locked decisions:

Props: `open`, `onClose`, `orderId`, `currentStatus`, `newStatus`, `customerEmail`, `onStatusChanged` callback

Content:

- Title: "Change Order Status"
- Description: `${STATUS_LABELS[currentStatus]} → ${STATUS_LABELS[newStatus]}`
- Email preview section: show subject line and brief body preview of what the customer would receive. Map transitions to email subjects:
  - pending → confirmed: "Your order has been confirmed"
  - confirmed → preparing: "Your order is being prepared"
  - preparing → out_for_delivery: "Your order is on its way"
  - out_for_delivery → delivered: "Your order has been delivered"
  - - → cancelled: "Your order has been cancelled"
- "Notify customer" checkbox — checked by default, with label showing customer email
- Reason text field — TextArea, required for cancellation (`newStatus === "cancelled"`), optional for others. Placeholder: "Reason for status change (required for cancellation)"
- Cancel / Confirm buttons with loading state

On confirm:

1. Call `PATCH /api/admin/orders/${orderId}/status` with `{ status: newStatus, notifyCustomer, reason }`
2. Call `onStatusChanged(newStatus)` for optimistic parent update
3. Close dialog

The parent (OrderDetailClient) handles optimistic UI:

- On status action button click: set `pendingStatus`, open dialog
- `onStatusChanged`: immediately update local order state to new status, trigger inline badge highlight animation (add a CSS class `animate-highlight` for ~1s then remove)
- On API failure: revert to previous status

Use the existing `ConfirmDialog` component from `@/components/ui/admin/settings/ConfirmDialog` as the base dialog, or build a custom one if ConfirmDialog doesn't support custom content (it may be simple confirm/cancel only). Check ConfirmDialog props — if it only has `description` string, build a richer dialog using the existing `Modal` component pattern from the codebase.
