  <files>
    src/app/api/admin/profile/route.ts

  <action>
Create `/api/admin/profile` route with GET and PATCH handlers.

**GET handler:**
1. `requireAdmin()` auth check
2. Fetch profile from `profiles` table: `id, email, full_name, phone, role, created_at`
3. Get auth user via `supabase.auth.getUser()` to extract:
   - `user.identities` array — map to provider names (e.g., "google", "email")
   - `user.app_metadata.role` — admin role
   - `user.created_at` — account creation date (use this for "Member since")
4. Return response:
```typescript
{
  data: {
    id: string,
    email: string,
    fullName: string | null,
    phone: string | null,
    role: string,  // from profiles.role or app_metadata.role
    authProvider: string,  // e.g., "google", "email" from identities[0].provider
    memberSince: string,  // user.created_at ISO string
    createdAt: string,  // profile created_at
  }
}
```

**PATCH handler:**
1. `requireAdmin()` auth check
2. Validate body with Zod: `{ fullName?: string, phone?: string | null }`
3. Update `profiles` table with new values + `updated_at` timestamp
4. Return updated profile in same shape as GET

Follow the pattern from `src/app/api/account/profile/route.ts` but add role and auth provider data. Do NOT allow role changes via PATCH — role is read-only here.

- Route compiles with GET and PATCH exports
  </verify>

- GET /api/admin/profile returns profile with role, authProvider, memberSince
- PATCH /api/admin/profile updates fullName and phone
- Role and auth provider are read-only (not updatable via PATCH)
  </done>

**1. Activity stats route** (`src/app/api/admin/profile/stats/route.ts`):

GET handler:
1. `requireAdmin()` auth check
2. Get auth user for `last_sign_in_at`
3. Count orders processed by this admin — query `order_audit_log` where `actor_role = 'admin'` and count distinct `order_id` entries. If actor identification is ambiguous (no actor_id column), count ALL admin-role audit entries as a team metric. Check the `order_audit_log` schema first.
4. Return:
```typescript
{
  data: {
    lastLoginAt: string | null,  // from user.last_sign_in_at
    ordersProcessed: number,     // count from audit log
  }
}
```

**2. Notification preferences route** (`src/app/api/admin/profile/notifications/route.ts`):

Reuse `customer_settings` table (per research recommendation — it's per-user, not per-role).

GET handler:
1. `requireAdmin()` auth check
2. Fetch from `customer_settings` where `user_id = admin's user ID`
3. Parse `notification_prefs` JSON column
4. Return with defaults if no row exists:
```typescript
{
  data: {
    orderConfirmation: true,
    orderCancellation: true,
    orderDelivered: true,
    newOrderAlert: true,
  }
}
```

PUT handler:
1. `requireAdmin()` auth check
2. Validate body with Zod schema matching the notification pref shape
3. Upsert to `customer_settings`: if row exists for user_id, update `notification_prefs`; if not, insert new row
4. Return saved preferences

Check `customer_settings` table schema in `src/types/database.ts` for exact column names. If the table has required columns beyond `user_id` and `notification_prefs`, provide sensible defaults.
