**Both routes -- Idempotency-Key handling:**

Add idempotency key checking at the START of each handler (after auth, before business logic):

1. Read header: `const idempotencyKey = request.headers.get('Idempotency-Key')`
2. If `idempotencyKey` is present:
   a. Check Supabase `idempotency_keys` table: `SELECT response_status, response_body FROM idempotency_keys WHERE key = $1 AND endpoint = $2`
   b. If found: return the cached response immediately (same status, same body). This prevents duplicate processing.
   c. If not found: proceed with normal processing, then AFTER successful response, INSERT into `idempotency_keys`:
      ```sql
      INSERT INTO idempotency_keys (key, endpoint, response_status, response_body, created_at)
      VALUES ($1, $2, $3, $4, NOW())
      ON CONFLICT (key) DO NOTHING
      ```
3. If no `idempotencyKey` header: proceed normally (backward compatible)

**IMPORTANT:** The `idempotency_keys` table may not exist yet. Since this is a small driver fleet and adding a DB migration adds complexity, use a **simpler approach**:

Instead of a DB table, use a lightweight approach:
- For the **stop status PATCH**: The server already validates status transitions (`isValidStatusTransition`). If a duplicate "mark as arrived" comes in and the stop is already "arrived", the transition validation returns 400 "Cannot transition from arrived to arrived". This is effectively idempotent -- the duplicate is rejected, and the client (which treats 4xx as permanent failure) removes it from the queue. **No additional code needed here for the common case.**
- The Idempotency-Key header is SENT by the client for future-proofing but **no server-side table is needed in this phase**. The status transition validation serves as the natural idempotency guard.

So the actual implementation is:
- **stop route.ts**: No changes needed. Status transition validation already prevents duplicate updates. Add a `// Idempotency: status transition validation prevents duplicate updates` comment near the `isValidStatusTransition` check.
- **exception route.ts**: The exception route checks `if (stop.status === "delivered" || stop.status === "skipped")` which prevents duplicate exceptions on already-completed stops. Add a comment: `// Idempotency: status check prevents duplicate exception reports for completed stops`. However, a rapid double-tap could create two exception records before the first marks the stop as skipped. To prevent this:
  - After `reportExceptionSchema.safeParse`, check for existing exception: `SELECT id FROM delivery_exceptions WHERE route_stop_id = $stopId LIMIT 1`. If found, return the existing exception ID with 200 (idempotent response).
  - Add this check before creating the new exception.
