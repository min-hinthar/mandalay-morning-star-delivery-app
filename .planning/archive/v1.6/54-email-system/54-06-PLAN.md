  <name>Task 1: Delivery reminder cron endpoint</name>
  <files>src/app/api/cron/delivery-reminders/route.ts</files>
  <action>
    Create `src/app/api/cron/delivery-reminders/route.ts`:

    **GET handler (cron-triggered):**
    1. **Auth guard:** Check for `Authorization: Bearer ${CRON_SECRET}` header. If CRON_SECRET env var is set, validate it. This prevents unauthorized access. Return 401 if invalid.

    2. **Query qualifying orders:**
       ```
       SELECT orders.id, orders.delivery_window_start, orders.delivery_window_end,
              orders.special_instructions,
              profiles.full_name, profiles.email, profiles.id as user_id,
              addresses.line_1, addresses.line_2, addresses.city, addresses.state, addresses.postal_code
       FROM orders
       JOIN profiles ON orders.user_id = profiles.id
       LEFT JOIN addresses ON orders.address_id = addresses.id
       WHERE orders.delivery_window_start::date = CURRENT_DATE
         AND orders.status IN ('confirmed', 'preparing')
       ```
       Use createServiceClient() for service role access.
       Also fetch order_items (name_snapshot, quantity) for each order to get item names and count.

    3. **Deduplicate:** Check notification_logs for existing delivery_reminder for each order_id today. Skip if already sent.

    4. **Send emails with staggering:**
       - For each qualifying order:
         - Import DeliveryReminder from '@/emails/DeliveryReminder'
         - Create DeliveryReminder element with: customerName, orderId, itemCount, itemNames (first 3), deliveryWindowStart, deliveryWindowEnd, address, specialInstructions, driverName (if assigned)
         - Call sendEmail() with: type 'delivery_reminder', idempotencyKey: `delivery-reminder-${orderId}-${today}`, mandatory: false (respects reminders preference)
         - Wait 100ms between sends to stay under Resend's 10/s rate limit
       - Track: sent count, skipped count (already sent), failed count

    5. **Same-day order handling (per discretion):** If delivery_window_start is today and the cron runs, send the reminder immediately (don't skip same-day orders).

    6. **Response:** Return JSON with { sent, skipped, failed, total }

    7. **Logging:** Log summary via logger.info with sent/skipped/failed counts.

    **Note:** The actual cron scheduling (Vercel cron, pg_cron, or external) is deployment-config. This endpoint just needs to be callable via GET with auth. Add a comment at top documenting the expected cron schedule: "Called daily at 8:00 AM PT. Configure via vercel.json cron or external scheduler."

<task type="auto">
  <name>Task 2: Resend webhook for email status tracking</name>
  <files>src/app/api/webhooks/resend/route.ts</files>
  <action>
    Create `src/app/api/webhooks/resend/route.ts`:

    **POST handler:**
    1. **Webhook signature verification:**
       - Resend sends a `svix-id`, `svix-timestamp`, `svix-signature` header trio
       - For MVP: verify using a shared RESEND_WEBHOOK_SECRET env var (simple bearer check in body, or svix verification if the svix package is available)
       - Simpler approach: check a custom `webhook-secret` header against RESEND_WEBHOOK_SECRET env var
       - If no RESEND_WEBHOOK_SECRET configured, skip verification but log warning

    2. **Parse event payload:**
       - Resend webhook events have: type, created_at, data (with email_id, to, subject, etc.)
       - Supported event types: 'email.delivered', 'email.opened', 'email.clicked', 'email.bounced', 'email.complained'

    3. **Map event to notification_logs status:**
       - email.delivered -> status: 'delivered'
       - email.opened -> status: 'opened'
       - email.clicked -> status: 'clicked'
       - email.bounced -> status: 'bounced'
       - email.complained -> status: 'bounced' (treat complaint as bounce)

    4. **Update notification_logs:**
       ```
       UPDATE notification_logs
       SET status = '{mapped_status}',
           metadata = jsonb_set(COALESCE(metadata, '{}'), '{resend_events}',
             COALESCE(metadata->'resend_events', '[]') || '[{"type": "{event_type}", "at": "{event.created_at}"}]')
       WHERE resend_id = '{event.data.email_id}'
       ```
       Use createServiceClient() for service role.

    5. **Return 200** immediately (Resend expects fast responses).

    6. **Error handling:** Log errors but always return 200 (prevent Resend retries on our processing errors).

  </action>

    - Maps all 5 Resend event types
    - Always returns 200
