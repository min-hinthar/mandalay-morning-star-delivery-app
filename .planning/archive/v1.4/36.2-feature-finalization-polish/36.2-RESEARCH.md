### Pattern 1: Resend Email Integration

**What:** Server-side email sending with React templates
**When to use:** Any transactional email (invites, notifications)
**Example:**

```typescript
// Source: Resend Node.js SDK documentation
import { Resend } from 'resend';
import { DriverInviteEmail } from '@/emails/DriverInviteEmail';

const resend = new Resend(process.env.RESEND_API_KEY);

export async function sendDriverInvite(email: string, token: string) {
  const { data, error } = await resend.emails.send({
    from: 'Morning Star <noreply@morningstardelivery.com>',
    to: email,
    subject: 'You\'re invited to join Morning Star Delivery',
    react: <DriverInviteEmail inviteUrl={`${BASE_URL}/driver/onboard/${token}`} />,
  });

  return { data, error };
}
```

### Anti-Patterns to Avoid

- **Building custom email service:** Use Resend, don't build SMTP integration
- **Storing tokens in localStorage:** Tokens go in database, not client
- **Auto-save forms:** CONTEXT.md specifies explicit save button required
- **Validation on blur:** CONTEXT.md specifies validation on submit only

## Don't Hand-Roll

| Problem          | Don't Build      | Use Instead                 | Why                                           |
| ---------------- | ---------------- | --------------------------- | --------------------------------------------- |
| Email delivery   | SMTP integration | Resend                      | Delivery tracking, templates, bounce handling |
| Email templates  | HTML strings     | React Email                 | Type-safe, reusable components                |
| Token generation | UUID only        | crypto.randomBytes          | More secure, less guessable                   |
| Form state       | useState         | React Hook Form             | Already in use, handles complex validation    |
| Skeleton loaders | Custom div       | Existing Skeleton component | Already implemented with Pepper theme         |

**Key insight:** Email is a solved problem. Resend handles delivery, bounces, and reputation. React Email makes templates maintainable. Don't build email infrastructure.

## Common Pitfalls

### Pitfall 1: Email Domain Verification

**What goes wrong:** Emails go to spam or fail silently
**Why it happens:** Resend requires domain verification for custom from addresses
**How to avoid:**

1. Verify domain in Resend dashboard
2. Add SPF/DKIM records
3. Use noreply@ for transactional emails
4. Include unsubscribe link (not required for transactional but good practice)
   **Warning signs:** Low delivery rate, no bounce notifications

### Pitfall 2: Token Expiration Edge Cases

**What goes wrong:** User clicks expired link, gets confusing error
**Why it happens:** Token checked at page load but form submission hours later
**How to avoid:**

1. Check token expiration at page load AND form submission
2. Show clear "Link expired" message with resend option
3. Include countdown timer on onboarding page
   **Warning signs:** Support tickets about "broken links"

### Pitfall 3: Optimistic UI Without Rollback

**What goes wrong:** UI shows success but server failed, data inconsistent
**Why it happens:** Missing error handling after optimistic update
**How to avoid:**

1. Store previous state before update
2. Wrap in try/catch with rollback
3. Show toast on error with "retry" option
   **Warning signs:** Users report "changes didn't save"

### Pitfall 4: Form Navigation Without Warning

**What goes wrong:** User loses unsaved changes
**Why it happens:** Missing beforeunload handler and in-app navigation guard
**How to avoid:**

1. Add beforeunload event listener (already in SettingsClient)
2. Add router navigation guard for in-app links
3. Show modal with Save/Discard/Cancel options
   **Warning signs:** User complaints about lost work

### Discard Changes Modal Pattern

```tsx
// Pattern for unsaved changes confirmation
// Source: Existing AlertDialog patterns in codebase
import {
  AlertDialog,
  AlertDialogAction,
  AlertDialogCancel,
  AlertDialogContent,
  AlertDialogDescription,
  AlertDialogFooter,
  AlertDialogHeader,
  AlertDialogTitle,
} from "@/components/ui/alert-dialog";

interface DiscardChangesModalProps {
  open: boolean;
  onDiscard: () => void;
  onSave: () => void;
  onCancel: () => void;
  isSaving: boolean;
}

export function DiscardChangesModal({
  open,
  onDiscard,
  onSave,
  onCancel,
  isSaving,
}: DiscardChangesModalProps) {
  return (
    <AlertDialog open={open}>
      <AlertDialogContent>
        <AlertDialogHeader>
          <AlertDialogTitle>Discard changes?</AlertDialogTitle>
          <AlertDialogDescription>
            You have unsaved changes. Would you like to save them before leaving?
          </AlertDialogDescription>
        </AlertDialogHeader>
        <AlertDialogFooter>
          <AlertDialogCancel onClick={onCancel}>Cancel</AlertDialogCancel>
          <AlertDialogAction variant="destructive" onClick={onDiscard}>
            Discard
          </AlertDialogAction>
          <AlertDialogAction onClick={onSave} disabled={isSaving}>
            {isSaving ? "Saving..." : "Save Changes"}
          </AlertDialogAction>
        </AlertDialogFooter>
      </AlertDialogContent>
    </AlertDialog>
  );
}
```
