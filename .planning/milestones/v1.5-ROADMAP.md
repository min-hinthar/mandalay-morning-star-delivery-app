# Milestone v1.5: Performance & Repo Health

**Status:** Planning
**Phases:** 40-47
**Primary Goal:** LCP < 2.5s (from current 8.1s)
**Secondary Goals:** Repo cleanup, large file refactoring, test coverage

## Overview

v1.5 addresses the JavaScript execution bottleneck causing 8.1s LCP by systematically reducing client bundle size and deferring non-critical code. The 5-phase LCP optimization follows a progressive strategy: quick wins first (image prioritization), then Server Component conversions, dynamic imports, provider refactoring, and finally animation optimization with CI guardrails. Repo health phases run in parallel after core LCP work, covering legacy doc archival and optional large file splits.

**Success Criterion:** LCP < 2.5s on mobile (Core Web Vitals "Good" rating)

## Phase Ordering Rationale

1. **Phase 40 first:** Quick wins require no code restructuring. Immediate measurable impact (2-3s reduction). Validates measurement approach.

2. **Phase 41 before Phase 42:** Server Components reduce overall bundle size, making dynamic import impact more visible. Establishes pattern for future work.

3. **Phase 42 before Phase 43:** Library-level optimization is lower risk than provider refactoring. Dynamic imports are well-documented, isolated changes.

4. **Phase 43 after core optimization:** Provider refactoring touches shared code. Earlier phases prove the value of bundle reduction, justifying the medium-risk refactor.

5. **Phase 44 last for LCP:** React Compiler and Lighthouse CI are infrastructure improvements. Set up after core optimization validates the approach.

6. **Phase 45 parallel-capable:** Repo cleanup can start after Phase 43, runs independently.

7. **Phase 46 optional:** Large file refactoring is maintenance improvement, not LCP-critical.
