## Context: What Was Just Fixed (2026-02-27)

### CI Errors Resolved

| Error | Root Cause | Fix Applied |
|-------|-----------|-------------|
| TS2345: `string[]` not assignable to `readonly OrderStatus[]` | `preDeliveryStatuses` in refund handler used `"ready"` (not a valid `OrderStatus`) and was typed as `string[]` | Typed as `OrderStatus[]`, removed invalid `"ready"` status |
| ESLint max-lines: 408 > 400 | `stripe/route.ts` grew past limit after 9-bug security fix session | Extracted 4 handler functions to co-located `handlers.ts` |
| Prettier violations in 3 files | Formatting drift from manual edits | Ran `prettier --write` on affected files |

### Bug Found During Fix

The `"ready"` status in `preDeliveryStatuses` was **not a valid `OrderStatus`** value. Valid statuses: `pending | confirmed | preparing | out_for_delivery | delivered | cancelled`. This means `.in("status", preDeliveryStatuses)` was including a non-existent status in the Supabase filter — functionally harmless but logically incorrect. Fixed by removing it.

**Decision needed:** Should `out_for_delivery` be considered a pre-delivery status for refund cancellation? Currently it is NOT — a full refund on an in-transit order preserves the order status. This may need a product decision.
